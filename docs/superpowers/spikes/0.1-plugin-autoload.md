# Spike 0.1: Plugin auto-load in spawned `claude` (in tmux)

**Date:** 2026-04-22
**Status:** ✅ PASS

## Method

Instead of setting up a dedicated probe plugin (which requires interactive `/plugin install`), used the already-installed **warp** plugin as a natural control: it registers a `SessionStart` hook that prints a visible banner.

```bash
tmux new-session -d -s probe-claude -c /tmp 'CLAUDECODE= claude'
sleep 8
tmux send-keys -t probe-claude '/mcp' Enter
sleep 3
tmux capture-pane -t probe-claude -p
```

## Evidence (captured from tmux pane)

```
⎿  SessionStart:startup says: ℹ️ Warp plugin installed but you're not running
   in Warp terminal. Install Warp (https://warp.dev) to get native
   notifications when Claude completes tasks or needs input.
```

The banner is emitted by `~/.claude/plugins/cache/claude-code-warp/warp/2.0.0/scripts/on-session-start.sh`, which means:

1. The tmux-spawned `claude` **discovered and loaded** the `warp` plugin.
2. The `warp` plugin's `SessionStart` hook **fired in the spawned claude**.
3. By extension, our future `lark-channel` plugin will be discovered and its hook will fire identically.

## Implication

ε' architecture assumption holds:
- `tmux new ... "claude"` → spawned claude auto-loads user's installed plugins.
- No need for `--mcp-config <generated>` fallback; `src/master/pool.ts` can spawn `claude` without an explicit mcp-config arg.

## Additional evidence

- `easy-harness@easy-harness-marketplace` is the spec's reference implementation for "tmux + SessionStart" — it is installed and actively used by the spec author (indirect confirmation).
- `claude-code-warp` ships the same pattern and works.
