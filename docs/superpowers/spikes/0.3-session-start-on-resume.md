# Spike 0.3: SessionStart hook behaviour on fresh start and `--resume`

**Date:** 2026-04-22
**Status:** ✅ PASS (fresh); ℹ️ informational (resume — matcher-dependent)

## Method

Observed the `warp` plugin's `SessionStart` hook behavior across two tmux-spawned claude scenarios:

### Fresh startup

```bash
tmux new-session -d -s probe-claude -c /tmp 'CLAUDECODE= claude'
```

**Result:** `SessionStart:startup says: ℹ️ Warp plugin installed…` banner appeared. Hook **fired**.

### Resume an existing session

```bash
tmux new-session -d -s probe-resume -c <workdir> \
  "CLAUDECODE= claude --resume 554fdbf7-5ff1-4b2a-a636-3b7f7a50e277"
```

**Result:** No warp banner shown. But the conversation transcript was fully restored.

## Explanation

Inspecting `~/.claude/plugins/cache/claude-code-warp/warp/2.0.0/hooks/hooks.json`:

```json
{
  "hooks": {
    "SessionStart": [
      { "matcher": "startup", "hooks": [ … ] }
    ]
  }
}
```

Warp's hook specifies `"matcher": "startup"` — it explicitly opts out of non-startup triggers. So the *absence* of a warp banner on resume does not prove the hook didn't fire; it proves the warp hook is `matcher`-filtered.

## Implication for `lark-channel`

Our `hooks/hooks.json` does **not** use a matcher, so it will fire on every `SessionStart` trigger (startup, resume, clear, compact, …). This matches spec §6.2's design:

- **Fresh start:** hook writes the new `session_id` into `by-id/<scopeId>.json` — the first-time capture.
- **Resume:** hook re-writes the same `session_id` (idempotent; no-op data-wise).
- **/clear:** hook overwrites with the new `session_id` generated by `/clear` — correct behavior for "user wants to start over in this scope".

Defensive note: easy-harness' hook explicitly guards `if (todo.claudeSessionId) process.exit(0)` to avoid overwriting on subsequent SessionStart triggers — a signal that the author **observed** SessionStart firing more than once per session (confirming `--resume` / `/clear` trigger the hook too).

## Conclusion

Fresh startup firing is **confirmed by direct observation**. Resume-path firing is strongly indicated by:
1. Claude Code's SessionStart hook supports a `matcher` field (only useful if there are multiple trigger subtypes).
2. Easy-harness' hook script defensively handles "already-set" cases.

This is sufficient evidence to build against. If resume turns out not to fire, the failure mode is benign (we keep the originally-captured session_id across resumes, which is still correct).
